# Claude Code Configuration

Permission rules and hooks for Claude Code (the `claude` CLI). Configuration is symlinked to
`~/.claude/settings.json` by `install/claude-code.sh`. Steering and skills are managed separately by
`install/ai-tools.sh`.

## Philosophy

- **Parity with other tools**: Deny lists mirror Kiro CLI and Cursor — same commands blocked, same
  files protected, same system paths restricted
- **Deny-first security**: Deny rules always win regardless of allow rules. Sensitive files blocked
  at multiple levels (permission rules + hooks)
- **Minimal friction for reads**: Allow all file reads without prompting; deny rules block sensitive
  files (`.env`, `.key`, `.pem`, credentials)
- **Parity with Kiro CLI write model**: Allow edits in project directories without prompting; deny
  rules block system paths
- **Cross-tool hooks**: Shared hook scripts work identically across Kiro CLI and Claude Code

## Directory Structure

```text
etc/claude-code/
└── settings.json    # Permissions (allow/deny) and hooks configuration
```

Additional files managed by `install/ai-tools.sh` (not stored here):

- `~/.claude/CLAUDE.md` — generated by concatenating steering docs from `etc/ai/steering/`
- `~/.claude/skills/` — symlink to `etc/ai/skills/`

## Configuration Model

### Permission Rules

Claude Code uses `allow`/`deny` arrays in `settings.json` with pattern matching:

- **`Bash(pattern)`** — shell command permissions
- **`Read(pattern)`** — file read permissions
- **`Edit(pattern)`** — file write permissions

Evaluation order is **deny → ask → allow**. The first matching rule wins. Unmatched operations
prompt for user approval.

### Pattern Syntax

- `*` matches any sequence of characters including spaces
- Paths support gitignore-style patterns: `./relative`, `**/recursive`, `~/home`
- `//path` for absolute filesystem paths

### Hooks

Two hook events configured:

- **`PreToolUse`** with `matcher: "*"` — runs `block-env-files.sh` on every tool call. Blocks any
  access to `.env` files by inspecting tool inputs and exiting `2` if found.
- **`PostToolUse`** with `matcher: "Bash"` — runs `audit-shell-commands.sh` asynchronously to log
  AWS and kubectl commands to `~/.local/share/ai-audit/command-audit.jsonl`.

### Steering Distribution

Steering is not configured in this directory. `install/ai-tools.sh` generates per-file rules in
`~/.claude/rules/` from shared steering sources in `etc/ai/steering/`. Files with `paths:`
frontmatter get conditional loading (only injected when Claude reads matching files). `CLAUDE.md` is
a minimal pointer file, not a concatenated dump.

## Best Practices Followed

These practices come from the
[official permissions documentation](https://docs.anthropic.com/en/docs/claude-code/permissions) and
[hooks documentation](https://docs.anthropic.com/en/docs/claude-code/hooks):

- **Deny rules for sensitive files** — `.env*`, `.key`, `.pem`, `credentials*` all blocked via
  `Read` deny patterns
- **System path protection** — `/etc/`, `/usr/`, `/bin/`, `/sbin/`, `/System/` blocked for `Edit`
- **Destructive command blocking** — `rm`, `sudo`, `kill`, `reboot`, mutating `gh` and `kubectl`
  commands denied
- **`git add .` blocked** — prevents accidental bulk staging
- **Hook-based enforcement** — `PreToolUse` hook provides defense-in-depth beyond permission rules
- **`async: true` on audit hook** — non-blocking logging that doesn't slow tool execution
- **`$schema` reference** — enables IDE autocomplete and validation for settings.json
- **Timeout on hooks** — 5s for blocking hook, 10s for async audit hook

### What Claude Code Does NOT Have (vs Kiro CLI)

- No agent/persona system (single mode)
- No hook framework as flexible as Kiro CLI (no `agentSpawn`, no `matcher` patterns beyond tool
  name)
- No knowledge base integration (no semantic search over indexed repos)
- No `allowedTools` concept (everything is allow/deny/prompt)
- No MCP server management in settings.json (configured separately in `~/.claude.json`)

## Intentional Deviations

- **Steering generated by install script** — rather than manually maintaining `~/.claude/rules/`,
  `install/ai-tools.sh` generates per-file rules from shared `etc/ai/steering/` sources with
  `paths:` frontmatter for conditional loading. `CLAUDE.md` is a minimal pointer file.
- **No project-level settings** — only user-level `~/.claude/settings.json` exists. Official docs
  recommend `.claude/settings.json` committed to git for team sharing. Acceptable for personal use.

## Finding Specific Information

- **What's allowed without prompting**: Check `permissions.allow` array in `settings.json`
- **What's blocked entirely**: Check `permissions.deny` array in `settings.json`
- **Hook scripts**: `etc/ai/hooks/block-env-files.sh` and `etc/ai/hooks/audit-shell-commands.sh`
- **Steering content**: Run `wc -l ~/.claude/CLAUDE.md` to check size; source files in
  `etc/ai/steering/`
- **MCP servers**: Configured in `~/.claude.json` (bootstrapped by `install/claude-code.sh`)
- **Skills**: Symlinked from `etc/ai/skills/` to `~/.claude/skills/`
