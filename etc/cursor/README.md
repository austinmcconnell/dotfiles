# Cursor Configuration

CLI permissions and MCP server configuration for Cursor. The CLI config is symlinked to
`~/.cursor/cli-config.json` by `install/cursor.sh`. Steering rules (`.mdc` files) are generated
separately by `install/ai-tools.sh`.

## Philosophy

- **Permission parity**: Deny/allow lists mirror Claude Code and Kiro CLI — same protections
  regardless of which tool is active
- **CLI-focused**: Only manages CLI agent permissions and MCP servers. IDE-specific settings
  (auto-run mode, sandbox, Privacy Mode) are managed through the Cursor UI
- **Single source of truth**: Steering rules generated from shared `etc/ai/steering/` sources rather
  than maintained independently

## Directory Structure

```text
etc/cursor/
├── cli-config.json    # CLI permissions (allow/deny rules)
└── mcp.json           # MCP server definitions
```

Additional files managed by `install/ai-tools.sh` (not stored here):

- `~/.cursor/rules/*.mdc` — generated from `etc/ai/steering/` with frontmatter

## Configuration Model

### Permission Rules

Cursor CLI uses `allow`/`deny` arrays with pattern matching similar to Claude Code:

- **`Shell(pattern)`** — shell command permissions
- **`Read(pattern)`** — file read permissions
- **`Write(pattern)`** — file write permissions

Deny rules take precedence over allow rules. Unmatched operations prompt for user approval.

### MCP Servers

Configured in `mcp.json` with stdio transport:

- **`command` + `args`** — local server process
- **`env`** — environment variables with `${env:VAR_NAME}` interpolation
- Project-level `.cursor/mcp.json` takes priority over global `~/.cursor/mcp.json` for same-name
  servers

### Steering Rules (.mdc files)

Not stored in this directory. `install/ai-tools.sh` generates `.mdc` files in `~/.cursor/rules/`
from shared steering sources. The `.mdc` format supports:

- **`alwaysApply: true`** — injected into every conversation
- **`globs`** — auto-attached when matching files are referenced
- **`description`** — agent reads this to decide relevance (on-demand loading)

## Best Practices Followed

These practices come from the [official rules documentation](https://docs.cursor.com/context/rules)
and [agent security documentation](https://docs.cursor.com/en/account/agent-security):

- **Deny-first evaluation** — comprehensive deny list blocks destructive operations
- **`.env` file blocking** — `Read(./.env*)`, `Read(**/.env*)`, and shell patterns
- **Credential file blocking** — `.key`, `.pem`, `credentials*` denied
- **System path protection** — `/etc/`, `/usr/`, `/bin/`, `/sbin/`, `/System/` blocked for writes
- **Destructive command blocking** — `rm`, `sudo`, `kill`, mutating `gh` and `kubectl` commands
- **`git add .` blocked** — prevents accidental bulk staging
- **Pinned MCP package versions** — `@aashari/mcp-server-atlassian-jira@3.3.0` (not `@latest`)
- **Env var interpolation for secrets** — `${env:ATLASSIAN_SITE_NAME}` syntax

### What Cursor Does NOT Have (vs Kiro CLI)

- No hooks system in CLI — cannot replicate `block-env-files.sh` enforcement
- No `disabled` flag for MCP servers — cannot version-control servers not always needed
- No knowledge base integration
- No agent/persona system (single mode)
- No audit logging capability

### What Cursor Has That Others Don't

- **IDE sandbox mode** — OS-level sandboxing (Seatbelt on macOS) for agent commands
- **`.cursorignore`** — blocks agent from reading files entirely (stronger than deny rules)
- **`.cursorindexingignore`** — excludes files from semantic index without blocking reads
- **Privacy Mode** — prevents data retention on Cursor servers
- **Team Rules** — org-wide enforced rules via dashboard (paid plans)

## Intentional Deviations

- **Global config only** — no project-level `.cursor/cli.json`. Dotfiles pattern provides one config
  for all projects. Would need project-level configs for team work.
- **No `.cursorignore` managed here** — IDE agent file blocking is project-specific and managed
  per-repo, not globally via dotfiles.
- **Steering generated by install script** — rather than manually maintaining `.cursor/rules/`,
  `install/ai-tools.sh` generates `.mdc` files from shared sources. Generated files are not visible
  in this directory.
- **No IDE settings managed** — auto-run mode, sandbox configuration, and Privacy Mode are managed
  through the Cursor UI rather than dotfiles.

## Finding Specific Information

- **What's allowed without prompting**: Check `permissions.allow` array in `cli-config.json`
- **What's blocked entirely**: Check `permissions.deny` array in `cli-config.json`
- **MCP servers**: `mcp.json` in this directory
- **Generated steering rules**: `~/.cursor/rules/*.mdc` (source: `etc/ai/steering/`)
- **IDE agent settings**: Cursor Settings > Agent (not managed by dotfiles)
- **Privacy Mode**: Cursor Settings > Privacy (not managed by dotfiles)
